# JS Reverse 错题本

## 把 Cookie 当成完整 env
- 现象：已经抓到登录 Cookie，但本地仍无法稳定复现请求，或缺少 `spaceId`、`spaceViewId`、页面上下文。
- 错误判断：认为只要拿到 Cookie，就已经恢复了最小可用运行环境。
- 正确做法：把 env 恢复视为多源联合问题，继续核对真实请求体、初始化接口、localStorage、运行态 store、pathname 等来源。
- 快速检查：当前结论是否只基于 Cookie？是否已经对照真实成功请求确认 header、body、空间上下文都齐全？
- 来源任务：`analysis/notion2api`
- 适用范围：登录态站点、AI/SaaS 类富客户端、存在空间或租户上下文的 Web 应用

## 用页面推断替代真实网络证据
- 现象：页面变量、源码搜索或 store 中能看到某字段，于是直接据此写脚本，但真实请求仍不一致。
- 错误判断：把页面可见值当成最终请求发送值。
- 正确做法：真实成功请求始终优先于页面推断；源码和页面变量只负责解释字段来源，不替代网络证据。
- 快速检查：当前字段来源有没有在真实请求头、真实请求体或初始化接口响应里被直接确认？
- 来源任务：`analysis/notion2api`
- 适用范围：动态 header、动态 body、运行时上下文拼装类任务

## 把字段缺失误判为字段不存在
- 现象：某些关键字段第一次抓取为空，于是怀疑字段名错误、接口变化或不存在。
- 错误判断：忽略了账号仍在 onboarding、页面仍半初始化、相关初始化接口尚未触发的可能性。
- 正确做法：先检查页面所处生命周期，再判断字段来源是否真的失效；必要时推进 onboarding 或进入真实工作空间后再抓。
- 快速检查：当前账号是否仍停在 onboarding？是否已经进入真实工作空间？相关初始化接口是否已经实际触发？
- 来源任务：`analysis/notion2api`
- 适用范围：新账号、首登账号、空间初始化依赖强的站点

## 默认 usage 或 billing 接口实时更新
- 现象：发送一次真实请求后，马上查看 usage / billing 接口，发现值没变，就误判抓错接口或脚本无效。
- 错误判断：默认服务端会在单次请求完成后立刻同步更新计量接口。
- 正确做法：把“是否实时更新”当成待验证事实，做请求前后差量对比，并接受异步结算、延迟刷新、批量累计等可能性。
- 快速检查：是否已经记录操作前后的接口基线？是否只验证了一次？是否把“不变”与“接口无价值”混为一谈？
- 来源任务：`analysis/notion2api`
- 适用范围：额度、积分、配额、订阅、账单类接口分析

## 过早坚持纯 HTTP 方案
- 现象：已经出现服务端明确拒绝非真实浏览器链路的证据，但仍长期停留在 requests/cloudscraper 路线上排障。
- 错误判断：把“还能发出请求”误当成“纯脚本仍有希望”，没有根据真实证据及时升级方案。
- 正确做法：先做证据化判断；若已确认服务端拒绝非真实浏览器链路，应及时切换到 Playwright 无头方案，再继续修指纹和自动化特征。
- 快速检查：是否已经抓到明确拒绝证据，例如登录校验失败、风控错误码、浏览器指纹差异？若有，是否还在无意义坚持纯 HTTP？
- 来源任务：`analysis/notion2api`
- 适用范围：存在登录、风控、设备指纹校验的 Web 站点
