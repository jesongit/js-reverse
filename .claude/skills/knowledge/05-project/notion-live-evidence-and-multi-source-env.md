# Notion 类站点的实时证据优先与多源 env 提取

## 适用场景

- 目标站点存在真实登录态、动态 Cookie、空间上下文或运行时生成请求体
- 目标不是单纯还原一个签名算法，而是恢复可稳定运行的完整请求环境
- 需要从浏览器中补 `.env`、Cookie、header、body、spaceId、spaceViewId` 等运行参数

## 核心结论

### 1. 真实网络请求优先级最高

当目标是恢复可运行请求，而不是只证明局部算法时，优先证据应是：

1. 真实网络请求
2. 初始化接口响应
3. localStorage / sessionStorage
4. 页面运行态 store
5. Cookie
6. 页面源码或脚本推断

不要把“页面变量存在”误当成“最终请求一定这么发”。真实请求头、真实请求体、真实上下文始终优先。

### 2. env 往往不是单源可恢复

对于 Notion 一类站点，最小可用环境通常来自多源联合：

- Cookie：身份与设备相关字段
- 请求体：spaceId、spaceViewId、thread 上下文
- 初始化接口：空间、用户、订阅、页面上下文
- localStorage：最近访问空间、视图、onboarding 状态
- 运行时 store：当前页面尚未落盘的空间信息
- URL/pathname：最后兜底的空间标识线索

如果只抓 Cookie，经常能“看起来已登录”，但实际仍无法稳定复现请求。

### 3. 先证据化，再决定交付形态

交付优先级仍应保持：

1. 纯脚本
2. Playwright 无头
3. Playwright 有头

但是否能停留在纯脚本，不应靠直觉判断，而要先看真实证据：

- 关键 header / body 是否可稳定还原
- 登录态是否可从单一凭证恢复
- 页面上下文是否必须依赖浏览器初始化
- 风控是否明确拒绝非真实浏览器链路

## 推荐分析顺序

### A. 先抓一条真实成功请求

至少确认：

- URL
- method
- 请求头中哪些字段不是固定值
- 请求体中哪些字段来自页面上下文
- 响应结构是否为 JSON、NDJSON、SSE、WebSocket 等

### B. 再追字段来源

字段来源排查顺序建议：

1. 请求体直接读值
2. 发起栈 / initiator
3. 初始化接口
4. 本地存储
5. 页面运行态对象
6. 源码搜索

### C. 最后判断是否需要浏览器交付

只有在真实证据已经证明：

- 非浏览器链路被服务端拒绝
- 或关键上下文无法脱离浏览器稳定获得

才升级到浏览器交付方案。

## 常见误判

### 误判 1：看见 Cookie 就以为 env 齐了

正确做法：继续核对 spaceId、spaceViewId、页面上下文、初始化接口依赖是否齐全。

### 误判 2：从源码推出字段来源后，就不再核对真实请求

正确做法：源码只用于解释，不替代真实请求证据。

### 误判 3：某字段首轮提取不到，就认为字段不存在

正确做法：先检查当前账号是否还停在 onboarding、是否还没进入真正工作空间、是否尚未触发相关初始化接口。

### 误判 4：usage / billing 接口会在一次请求后立即同步更新

正确做法：把“是否实时更新”当成待验证事实，用真实前后对比确认，不要默认即时一致。

## 快速检查清单

- 是否已经抓到一条真实成功请求？
- 当前结论是来自真实网络还是页面推断？
- env 是否来自多源联合，而不是只看 Cookie？
- 当前账号是否仍卡在 onboarding 或半初始化状态？
- 如果某接口值没变，是否已经做过前后差量验证？
